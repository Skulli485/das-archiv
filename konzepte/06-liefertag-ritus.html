Der Liefertag-Ritus

Wonach sieht eine fertige Arbeit aus, wenn sie nirgendwo veröffentlicht wird? Und warum reicht der Beweis auf dem eigenen Rechner vollkommen aus?

Die ganze Woche über ist ein kleines, senkrechtes System gewachsen: eine Vue-Galerie vorne, dahinter der Lesetisch aus Konzept 01 als Grenze zur Außenwelt, auf dem Tisch ein Gedächtnis, das schon geholte Karten wiedererkennt, und ganz hinten der Tresor mit der echten, fremden API. Der Liefertag baut diesem Haus kein weiteres Stockwerk an. Er ist der Tag, an dem jemand einmal durch das ganze Haus geht und prüft, ob es trägt.

Diese Prüfung braucht keine neue Adresse im Netz. Sie braucht deinen eigenen Rechner, dein eigenes Terminal, und eine feste Reihenfolge von Schritten, die jeden Teil des Systems einzeln unter Druck setzt.

Kernsatz

„Liefertag heißt fertigstellen und beweisen, nicht deployen." Kein Hosting, kein Vercel, keine Pages. Der Beweis ist lokal: dein Rechner, dein Terminal, deine eigene Preview.

§1Fertigstellen ist nicht deployen

Ein Liefertag fühlt sich an wie ein Anlass, etwas zu veröffentlichen. Genau das ist er hier nicht: Es gibt keine Adresse im Netz, die deinen Namen trägt. Was zählt, ist, ob du in wenigen Minuten, auf deinem eigenen Rechner, zeigen kannst, dass jede Zusage aus der Woche hält, dass gebaut wird, dass die Grenze aus Konzept 01 ohne CORS steht, dass der Cache Karten wiedererkennt, dass die Galerie ohne Maus bedienbar bleibt.

Jemand, der dein Repository klont, durchläuft dieselbe Reihenfolge von Beweisen, die du gerade selbst durchgehst. Genau darin liegt der Unterschied zwischen Liefertag und gewöhnlichem Werktag: nicht mehr Code, sondern ein Rückblick auf das, was schon da ist.

§2Der Ritus in sieben Schritten

Jeder Schritt prüft einen anderen Teil des Systems und liefert einen eigenen, kleinen Beweis:

  1. bun run build erzeugt das Produktions-Artefakt dist/. Beweist: das Projekt baut ohne Fehler, aus dem echten Quellcode, nicht nur im Kopf.
  2. bun run preview (bzw. vite preview) serviert genau dieses dist/ lokal. Der Proxy auf /api läuft auch hier, also spricht die Preview mit dem Bun-Server ohne CORS, vorausgesetzt der Server läuft (cd server, dann bun start). Beweist: das gebaute Artefakt funktioniert, nicht nur der bequeme dev-Server.
  3. Lighthouse läuft in den Browser-DevTools gegen diese lokale Preview. Ziel: Leistung ≥ 95. Weil das Bundle klein bleibt (etwa 60-70 kB JavaScript, gzip rund 25 kB), ist dieses Ziel ohne Tricks erreichbar. Beweist: die Seite ist tatsächlich schnell, nicht nur gefühlt schnell.
  4. Die 390px-Kontrolle: DevTools-Geräteansicht auf 390px Breite. Galerie und Detailkarte sitzen ohne horizontales Scrollen. Beweist: das Layout hält auf einem schmalen Telefon, nicht nur auf dem breiten Monitor.
  5. Der Tastatur-Durchgang: nur mit Tab und Enter bzw. Leertaste durch die Galerie. Ein sichtbarer Fokusring, jede Kachel fokussierbar und auslösbar. Beweist: die Galerie funktioniert auch ohne Maus.
  6. git log --oneline zeigt die Historie. Sie ist der Faden der Arbeit: ein Commit pro Stufe, nicht ein einziger Riesen-Commit am Ende. Beweist: der Weg zur Lösung war nachvollziehbar, nicht nur das Ergebnis.
  7. Die README beschreibt, wie man beide Prozesse startet, den Hinweis auf Redis unter Port 6380, und den Hinweis auf das Degrade-Verhalten, falls der Cache einmal fehlt. Beweist: ein fremder Rechner kommt ohne Rückfragen in denselben Zustand wie deiner.
Falle

Lighthouse misst nicht gegen den dev-Server. Dort ist der Code unminifiziert und langsamer, das Ergebnis verzerrt in beide Richtungen. Gemessen wird gegen die preview, das echte gebaute Artefakt. Und: kein Deploy, der Ritus endet lokal, nicht auf einer fremden Domain.

Zwei Terminals, der Ritus in Befehlen 
# Terminal 1, der Server bleibt während der Prüfung an
cd server
bun start

# Terminal 2, bauen, dann gegen das echte Artefakt prüfen
bun run build
# dist/ entsteht, keine rote Zeile ist der erste Beweis

bun run preview
# serviert dist/ lokal, Proxy /api aktiv, kein CORS
# jetzt: Lighthouse in den DevTools gegen genau diese Adresse

§3Die Qualitäts-Checkliste vor der Abgabe

Jeder Beweis aus dem Ritus lässt sich zeigen, nicht nur behaupten:

BeweisWie du ihn zeigst
Build ohne Fehler Terminal-Ausgabe von bun run build, ohne rote Zeilen
Leistung ≥ 95 Lighthouse-Screenshot gegen die Preview
Kein Layout-Sprung feste width/height auf jedem Bild (kein CLS)
Bilder laden spät nach loading="lazy" auf den Galerie-Bildern
Erreichbar per Tastatur sichtbarer Fokusring beim Tab-Durchgang
Bewegung wird respektiert prefers-reduced-motion greift, Animationen beruhigen sich

Ein paar kleine technische Details tragen diese Beweise mit, sichtbar in jeder Code-Review:

  • loading="lazy"
  • feste width/height (kein CLS)
  • alt = Titel + Künstler
  • sichtbarer Fokusring
  • prefers-reduced-motion
  • tabular-nums für ms
