Das Archiv · Konzept 04

Graceful Degradation

Was passiert mit deinem Cache, wenn Redis gar nicht da ist? Und warum merkt der Leser vorne am Tisch davon nichts?

Aus Konzept 03 kennst du das Cache-Aside-Muster mit TTL: Der Lesetisch merkt sich eine Katalogkarte für eine Weile, damit er nicht bei jeder Anfrage erneut in den Tresor muss. Offen blieb dabei, wo genau diese Karte liegt, solange sie gültig ist. Es gibt zwei mögliche Orte. Der erste ist ein gemeinsamer Karteikasten im Keller des Hauses, auf den jeder Lesetisch im Gebäude zugreifen kann. Der zweite ist eine gewöhnliche Schublade direkt an diesem einen Tisch, die niemand sonst kennt.

geteilt, externKarteikasten im Keller
→
gleiche Schnittstelleget / set / ttl
←
privat, lokalSchublade am Tisch

Redis trägt hier noch einmal die Farbe des Tresors, nicht weil es dieselbe fremde API wäre, sondern weil es denselben Charakter hat: außerhalb dieses einen Prozesses, geteilt mit anderen, nur über das Netz erreichbar. Die Schublade bleibt bei der Farbe des Tisches: direkt hier, direkt jetzt, nur diesem einen Prozess bekannt. Für den Leser vorne am Tisch macht das keinen Unterschied. Er bekommt in beiden Fällen dieselbe Katalogkarte, in derselben Form, ungefähr gleich schnell.

§1Eine Schnittstelle, zwei Speicherorte

Der Rest des Servers kennt den Cache nicht als „Redis" oder „Map". Er kennt ihn nur als eine winzige Schnittstelle mit drei Verben: eine Karte holen, eine Karte ablegen, nachsehen, wie lange sie noch gültig ist.

Die gesamte Schnittstelle, mit der der Rest des Servers arbeitet:

  • get(key)
  • set(key, value, ttl)
  • ttl(key)

Hinter dieser Schnittstelle sitzt entweder ein Redis-Client oder eine gewöhnliche JavaScript-Map. Beide füllen dieselben drei Verben mit Leben, beide liefern dieselbe Form zurück. Weil die Schnittstelle identisch bleibt, ändert sich für den Rest des Codes: nichts. Kein Aufrufer fragt jemals, ob Redis läuft. Er fragt nur nach dem Wert zu einem Schlüssel.

Warum das für die Hausaufgabe zählt

Redis ist für diese Aufgabe deshalb optional. Ohne Docker läuft derselbe Server, mit derselben Katalogkarte, im selben Tempo, nur ein Detail im Vertrag wechselt: source zeigt dann "map" statt "redis". Der Rest der Antwort, { source, ms, ttl, data }, bleibt, wie er war.

§2Der Testtresor an Port 6380

Willst du den Redis-Modus wirklich sehen, brauchst du einen laufenden Redis-Server. Nicht deinen echten, falls irgendwo einer für ein anderes Projekt läuft, und nicht den Standardport 6379. Für diese Aufgabe reicht ein Wegwerf-Container auf Port 6380, gestartet mit einer einzigen Zeile.

Terminal, der Testtresor, ein Kommando 
docker run -d --rm -p 6380:6379 --name archiv_redis redis:alpine

-d schickt den Container in den Hintergrund, --rm räumt ihn beim Stoppen restlos weg, -p 6380:6379 bildet deinen lokalen Port 6380 auf den internen Redis-Port 6379 im Container ab. Dieser Redis kennt keine wichtigen Daten. Ihn zu verlieren, neu zu starten oder einfach zu vergessen, kostet nichts.

Aus dem Tresor

6380, nie 6379, absichtlich. So kollidiert dein Testcontainer nie mit einem echten Redis, der auf demselben Rechner vielleicht schon für ein anderes Projekt läuft. Taucht dieser Container nicht in docker ps auf, bist du automatisch im Map-Modus, noch bevor der Server überhaupt ein Zeitlimit braucht.

§3Wenn der Tresor nicht antwortet: das Zeitlimit

Jetzt der Teil, an dem die meisten ersten Versuche stolpern. Was passiert, wenn dein Server versucht, sich mit Redis zu verbinden, aber Port 6380 gerade niemand offen hält, weil der Container nicht läuft oder noch nicht bereit ist?

Die Falle: connect() hängt

Der naheliegende Code sieht harmlos aus: await redis.connect(), dann weiter zum nächsten Schritt. Ist der Port aber zu, bricht Buns eigener Redis-Client diesen Aufruf nicht sofort mit einem Fehler ab. Er hängt, reglos, ohne Fehler und ohne Antwort. In einem Testlauf blockierte ein ungesicherter connect() rund zwei Minuten, bevor überhaupt etwas zurückkam. In dieser Zeit steht der gesamte Server still, nicht nur die Cache-Anfragen.

Die Lösung ist kein Sonderfall für Redis, sondern ein allgemeines Muster: Jeder Redis-Aufruf bekommt einen Konkurrenten, einen Timer. Wer zuerst antwortet, gewinnt. Antwortet Redis nicht rechtzeitig, gewinnt der Timer, und du weichst aus, statt zu warten.

mitZeitlimit, der Wettlauf gegen das Hängen 
// Zwei Versprechen treten gegeneinander an, nur das schnellere zählt.
function mitZeitlimit(promise, ms, tag) {
  const zeitlimit = new Promise((_, ablehnen) =>
    setTimeout(() => ablehnen(new Error(`Zeitlimit: ${tag}`)), ms)
  )
  return Promise.race([promise, zeitlimit])
}

// Beim Boot: entweder antwortet Redis rechtzeitig, oder wir weichen aus.
try {
  await mitZeitlimit(redis.connect(), 1500, 'connect')
  // ab hier: Redis-Modus
} catch (fehler) {
  console.log('[cache] Redis nicht erreichbar (...) -> In-Memory Map')
  // ab hier: Map-Modus
}

Verliert Redis das Rennen, ganz gleich ob durch Zeitlimit oder durch einen echten Fehler, greift genau eine Reaktion: zurück auf die Schublade, so simpel wie hier skizziert.

Skizze, die Schublade, wenn kein Keller da ist 
// Kein Redis nötig: eine gewöhnliche Map hält Wert und Ablaufzeit.
const speicher = new Map()
speicher.set(schluessel, { wert, ablaeuftAm: Date.now() + ttlSekunden * 1000 })

Genau eine Protokollzeile, nicht mehr. Kein Stacktrace, keine Warnung bei jeder einzelnen Anfrage danach. Der Server trifft die Entscheidung einmal beim Boot und hält sich daran. Die vollständige initCache() mit beiden Backends baust du in Aufgabe 04.

§4Zwei Modi, ein Verhalten

Am Ende gibt es zwei fertige Zustände, in die dein Server booten kann. Beide erfüllen dieselbe Schnittstelle, beide liefern die Katalogkarte im selben { source, ms, ttl, data }-Vertrag aus Konzept 01. Der Unterschied liegt allein darin, was hinter den drei Verben steckt.

Vom Tisch aus gesehen

Graceful Degradation heißt nicht, dass nichts ausfallen darf. Es heißt, dass das System bei einem fehlenden Teil leiser wird, nicht kaputt. Fehlt der Karteikasten im Keller, springt die Schublade ein, und der Leser verliert keine Karte. Er verliert höchstens die Gewissheit, dass sie einen Neustart überlebt hätte, und selbst das sieht er nur, wenn er das kleine Badge neben der Karte liest.

Die Frage des Tages

„Startet dein Server gerade im Redis-Modus oder im Map-Modus, und woran erkennst du das, ohne eine einzige Zeile Code zu lesen?"

Mitnehmen

Eine Schnittstelle trennt, was der Rest des Codes wissen muss, von dem, was dahinter passiert. Redis oder Map ist eine Entscheidung, die einmal beim Boot fällt, mit einem Zeitlimit gegen das Hängen abgesichert, und danach nur noch in einer einzigen Protokollzeile laut wird. Der Leser am Tisch bekommt in beiden Fällen dieselbe Katalogkarte.
