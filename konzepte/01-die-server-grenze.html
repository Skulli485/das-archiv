Die Server-Grenze

Warum steht ein eigener Server zwischen deinem Browser und einer fremden, langsamen API? Und warum verschwindet damit der CORS-Schmerz von selbst?

Stell dir einen Leseraum vor. Tief im Haus liegt der Tresor: eine riesige, langsame Sammlung, weit weg, die auf jede Anfrage ein rohes Blatt mit fünfzig Feldern herausgibt. Kein Besucher darf selbst hinein. Stattdessen sitzt am Eingang der Lesetisch. Du sagst ihm eine Nummer, er geht in den Tresor, holt das Blatt, schreibt eine saubere Katalogkarte ab und legt sie dir hin. Nur die Karte überquert den Tisch, nie das rohe Blatt.

LeserBrowser (Vue)
→
LesetischDein Bun-Server
→
TresorFremde API

Dieser Lesetisch ist die Grenze. Er ist der einzige Ort, an dem dein System die fremde Welt berührt. Alles, was langsam, unbeständig oder unbequem geformt ist, bleibt hinter ihm. Nach vorne, zum Browser, gibt er einen klaren, ruhigen Vertrag heraus.

§1Warum überhaupt eine Grenze?

Der Browser könnte die fremde API auch direkt rufen. In der Theorie. In der Praxis holst du dir damit drei Probleme gleichzeitig ins Haus:

ProblemOhne GrenzeMit Grenze (Server)
Form Der Browser bekommt 50 rohe Felder und muss selbst aufräumen. Der Server normalisiert einmal auf 7 saubere Felder.
Tempo Jeder Nutzer wartet bei jedem Klick die volle Sekunde. Der Server merkt sich die Antwort und liefert sie beim nächsten Mal sofort.
Bindung Aendert die fremde API ihre Felder, bricht dein Frontend. Nur eine Funktion am Server passt sich an, der Vertrag bleibt.
CORS Der Browser blockiert die fremde Antwort oft komplett. Kein Thema mehr, dazu gleich mehr.
Kernsatz

Eine Grenze ist keine zusätzliche Schicht aus Bequemlichkeit. Sie ist der Ort, an dem du entscheidest, was in dein System darf und in welcher Form. Wer diese Entscheidung nicht trifft, trifft sie trotzdem, nur verstreut über das ganze Frontend.

§2Der CORS-Schmerz · und warum er hier verschwindet

CORS steht für Cross-Origin Resource Sharing. Es ist eine Regel des Browsers, kein Gesetz des Internets. Der Browser sagt: „Diese Seite kommt von localhost:5173. Sie will Daten von fremde-api.de holen. Das ist eine andere Herkunft (Origin). Erlaubt die fremde Seite das ausdrücklich per Antwort-Kopfzeile? Wenn nicht, blockiere ich."

Der entscheidende Punkt: Nur der Browser prüft das. Ein Server, der von sich aus eine andere Adresse ruft, kennt diese Regel nicht. Für ihn ist ein fetch nach außen ein ganz normaler Aufruf.

gleiche HerkunftBrowser → /api
→
kein CORSServer → fremde API

Genau das nutzen wir aus. Der Browser spricht ausschließlich mit /api, also mit der eigenen Herkunft. Im Entwicklungsbetrieb leitet Vite jeden /api-Aufruf per Proxy an den Bun-Server weiter. Aus Sicht des Browsers bleibt alles zu Hause. Der Server wiederum ruft die fremde API, und dort gibt es kein CORS, weil kein Browser beteiligt ist.

vite.config.js, der Proxy, der alles zu Hause hält 
// Der Browser sieht nur /api. Vite reicht es an den Server auf :3000 weiter,
// im dev-Server UND in der preview. Darum nie CORS.
const proxy = { '/api': 'http://localhost:3000' }

export default defineConfig({
  plugins: [vue()],
  server: { proxy },
  preview: { proxy },
})
Aus dem Tresor

Weil derselbe Proxy in server (dev) und preview steht, läuft dein gebautes dist/ später genauso ohne CORS, und Lighthouse kann trotzdem gegen die lokale Preview messen.

§3Was überquert die Grenze? Der Vertrag

Die Grenze ist nur so gut wie das, was sie herausgibt. Der Server antwortet auf jede Objekt-Anfrage nicht mit dem rohen Blatt, sondern mit einem festen, kleinen Vertrag. Diese Form ist das Versprechen an das Frontend:

Die Antwort der Grenze, immer diese Form 
{
  source: "netz" | "cache",   // woher kam es?
  ms: 631.5,               // wie lange hat es gedauert?
  ttl: 60,                  // wie lange bleibt es gültig?
  data: { /* die Katalogkarte, 7 Felder */ }
}

Das Frontend muss nie wissen, ob die Karte frisch aus dem Tresor kam oder schon auf dem Tisch lag. Es bekommt immer dieselbe Form. Genau das macht die Grenze nachvollziehbar: sie sagt mit source und ms, woher und wie schnell, aber die Form bleibt gleich.

Die Frage des Tages

„Wer besitzt diese Grenze, und was überquert sie?"

Trage diese Frage durch die ganze Aufgabe. Jedes Mal, wenn du unsicher bist, wo etwas hingehört: Frag, auf welcher Seite der Grenze es lebt. Die rohen 50 Felder? Hinter der Grenze. Die 7 Felder der Katalogkarte? Davor. Der langsame Netz-Aufruf? Hinter der Grenze. Das {source, ms, ttl}? Genau auf der Grenze, als ihr Laufzettel.

Mitnehmen

Der Server ist eine Grenze, die die langsame, fremde Welt besitzt, nach vorne einen sauberen Vertrag herausgibt, und (Konzept 03) sich merkt, was sie schon einmal geholt hat. CORS ist dabei kein Feind, den du besiegst, sondern ein Problem, das durch die Grenze gar nicht erst entsteht.
