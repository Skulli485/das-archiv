Das Archiv · Konzept 03

Cache-aside & TTL

Warum ist der zweite Klick auf dasselbe Objekt um ein Vielfaches schneller als der erste? Und wie lange darf sich der Tisch auf eine Karte verlassen, bevor er noch einmal in den Tresor muss?

In Konzept 01 stand der Lesetisch zwischen dir und dem Tresor: du nennst eine Nummer, der Tisch holt das Blatt, schreibt eine Katalogkarte ab und reicht sie dir. Jedes Mal derselbe Weg, jedes Mal die volle Wartezeit. Ein guter Lesetisch merkt sich aber, was er schon einmal abgeschrieben hat. Legt jemand dieselbe Nummer noch einmal vor, greift der Tisch zur liegen gebliebenen Karte, statt erneut in den Tresor zu laufen. Genau das ist cache-aside: dein Server prüft zuerst das eigene Gedächtnis, bevor er die langsame, fremde Welt überhaupt anspricht.

§1Das Prinzip: erst schauen, dann holen

Der Name cache-aside beschreibt genau diese Reihenfolge. Der Cache liegt neben dem eigentlichen Weg zur Quelle, nicht davor verschmolzen. Deine Anwendung entscheidet bei jeder Anfrage selbst: zuerst auf den Tisch schauen. Liegt die Karte dort, ist die Sache erledigt. Liegt sie nicht dort, geht der Tisch in den Tresor, holt sie, schreibt sie sauber ab, und legt die Kopie diesmal zusätzlich auf den Tisch, für alle, die danach fragen. Ein anderer Name für dasselbe Muster ist lazy loading: der Tisch füllt sich nicht im Voraus, sondern erst durch echte Nachfrage.

Kernsatz

Cache-aside ist kein Trick am Tresor. Der Tresor weiß nichts von einem Cache. Die Entscheidung liegt vollständig bei deinem Server: erst fragen, ob die Antwort schon irgendwo liegt, und nur im Ernstfall den langsamen Weg gehen.

§2Der Lesealgorithmus

Vier Schritte, immer in derselben Reihenfolge. Der Anfang ist unabhängig davon, ob am Ende ein Treffer steht oder nicht:

  1. Einen Schlüssel bauen, der das Objekt eindeutig und stabil benennt, zum Beispiel archiv:objekt:436535.
  2. Mit diesem Schlüssel den Tisch fragen: cache.get(key).
  3. Treffer (Hit): die Karte liegt bereits da. Zurück geht sofort { source: "cache", ms, ttl, data }, schnell.
  4. Kein Treffer (Miss): erst jetzt in den Tresor. Die Antwort wird normalisiert und per cache.set(key, karte, ttl) auf dem Tisch abgelegt, dann geht die Antwort hinaus: { source: "netz", ms, ttl, data }, langsam, aber nur dieses eine Mal.
Skizze, der Lesealgorithmus als Pseudocode 
async function leseObjekt(id) {
  const key = `archiv:objekt:${id}`       // stabil, eindeutig, mit Namensraum
  const start = performance.now()

  const treffer = await cache.get(key)
  if (treffer) {
    return { source: "cache", ms: performance.now() - start, ttl: 60, data: treffer }
  }

  // Miss, erst hier wird der Tresor überhaupt bemüht
  const roh = await holeVomTresor(id)
  const karte = normalisiere(roh)          // die kleine Katalogkarte, nicht das rohe Blatt
  await cache.set(key, karte, 60)      // TTL in Sekunden

  return { source: "netz", ms: performance.now() - start, ttl: 60, data: karte }
}

„Die vollständige Cache-aside-Funktion baust du in Aufgabe 04."

§3Erst messen, dann cachen

Der Laufzettel { source, ms, ttl, data } aus Konzept 01 hat genau hier seinen Zweck: ms misst mit performance.now(), wie lange die Antwort tatsächlich gebraucht hat. Ohne diese Zahl ist ein Cache nur eine Behauptung. Mit ihr wird der Unterschied sichtbar, bevor überhaupt irgendetwas optimiert wird.

Ruf dasselbe Objekt zweimal auf. Der erste Klick ist spürbar langsam, weil er wirklich zum Tresor läuft:

631.5 ms

Der zweite Klick trifft auf die liegen gebliebene Karte:

1.8 ms · TTL 60s

Größenordnung: Netz liegt hier bei einigen hundert Millisekunden, Cache bei ein bis zwei, grob das 300- bis 600-fache. Diese Regel merkst du dir besser als jede einzelne Zahl: erst messen, dann cachen. Wer optimiert, ohne vorher gemessen zu haben, cacht am Ende oft genau die Stelle, die ohnehin schon schnell war.

§4TTL: das Verfallsdatum der Karte

Eine Karte, die für immer auf dem Tisch liegen bleibt, ist kein Cache mehr, sondern eine Behauptung über die Vergangenheit. Die TTL (time to live) legt fest, wie lange eine Karte als gültig gilt, bevor der Tisch sie verwirft und beim nächsten Aufruf wieder frisch aus dem Tresor holt.

Aus dem Tresor

Kurze TTL heißt: die Karte ist selten veraltet, aber der Tisch läuft häufiger in den Tresor, mehr Wartezeit für mehr Frische. Lange TTL heißt: fast immer schnell, aber die Karte kann eine Weile nicht mehr stimmen. Für diese Aufgabe reichen 60 Sekunden, lang genug, um den Effekt beim zweiten Klick zu sehen, kurz genug, um nicht ewig auf einer veralteten Karte sitzen zu bleiben. Das ist die kleine, klare Ausgabe eines größeren Problems: zu wissen, wann eine Kopie nicht mehr gilt, gehört zu den hartnäckigsten Fragen im Umgang mit Zustand überhaupt.

§5Zwei Wege durch denselben Tisch

Miss und Hit laufen durch dieselbe Funktion, aber nicht denselben Weg. Beim Miss macht der Tisch einen Umweg über den Tresor und legt danach ab, was er geholt hat:

LeserBrowser (Vue)
→
Lesetischcache.get → Miss
→
TresorFremde API
→
Lesetischcache.set, TTL 60s
→
LeserBrowser (Vue)

Beim Hit bleibt der Tresor außen vor, der Tisch antwortet allein:

LeserBrowser (Vue)
→
Lesetischcache.get → Hit
→
LeserBrowser (Vue)

§6Eine Falle: was genau liegt auf dem Tisch?

Der Cache ist nur so sauber wie das, was du in ihn legst.

Falle

Cache niemals das rohe Blatt aus dem Tresor, sondern die normalisierte Katalogkarte. Wer die volle Antwort mit fünfzig Feldern in den Cache legt, verschiebt das Formproblem aus Konzept 01 nur nach hinten, jeder, der später aus dem Cache liest, sieht wieder die rohe Form. Außerdem ist ein Schlüssel wie 436535 allein zu knapp: ohne Namensraum (archiv:objekt:436535) kollidiert er irgendwann mit einem Schlüssel aus einem ganz anderen Teil deiner Anwendung.

Die Frage des Tages

„Was genau liegt auf dem Tisch, und wie lange darfst du ihm noch trauen?"

Mitnehmen

Cache-aside heißt: deine Anwendung fragt zuerst das eigene Gedächtnis, bevor sie die langsame, fremde Welt bemüht. Der Laufzettel aus Konzept 01 macht diesen Unterschied sichtbar, source und ms sagen genau, welcher der beiden Wege gerade gelaufen ist. Und die TTL erinnert daran, dass jede gespeicherte Antwort ein Verfallsdatum braucht, sonst ist sie irgendwann keine Kopie mehr, sondern nur noch eine alte Behauptung. Was aber, wenn selbst der Tresor nicht mehr antwortet? Genau davon handelt Konzept 04: Graceful Degradation.
