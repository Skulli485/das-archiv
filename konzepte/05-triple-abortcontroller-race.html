Das Archiv · Konzept 05

Das Triple & die Wächter

Eine Anfrage ist niemals nur da oder nicht da. Sie hat einen Zustand mit vier Gesichtern, und zwei Wächter, die dafür sorgen, dass am Ende wirklich die richtige Antwort auf der Katalogkarte steht.

Konzept 01 hat die Grenze gebaut: dein Server als Lesetisch, der aus dem Tresor holt und eine saubere Katalogkarte zurückgibt, in einem festen Vertrag aus source, ms, ttl und data. Diese Karte überquert die Grenze genau einmal. Was danach mit ihr passiert, entscheidet sich nicht mehr am Tisch, sondern im Browser, in genau der composable, die du in Aufgabe 03 baust.

Zwischen „ich will die Karte" und „ich habe die Karte" liegt eine Zeitspanne. Und in dieser Zeitspanne kann viel passieren: der Nutzer wartet, klickt weiter, verlässt die Seite. Eine belastbare composable hält nicht nur den guten Fall aus, in dem genau eine Anfrage losläuft und brav zurückkommt. Sie hält auch den Fall aus, in dem zwei Anfragen gleichzeitig unterwegs sind.

§1Das Triple, das eigentlich ein Quadrupel ist

Der naheliegende erste Versuch ist ein einzelnes Flag: isLoading, daneben vielleicht noch data und error. Das wirkt einfach, und ist trotzdem ungenau. Ein Boolean kennt nur zwei Zustände. Er kann nicht sagen: „Ich wurde noch nie gefragt." Und er kann nicht gleichzeitig sagen: „Ich bin gerade dabei, aber der letzte Versuch ist gescheitert."

Die klarere Lösung ist ein einzelner, benannter zustand mit genau vier möglichen Werten. Kein Kombinieren mehrerer Booleans, kein Raten daraus, ob data === null nun „noch nicht geladen" oder „leer zurückgekommen" bedeutet.

  • idle
  • laden
  • fertig
  • fehler
ZustandWannWas der Nutzer sieht
idle Vor dem ersten Aufruf, noch nie gefragt. Ein stiller Hinweis, keine Karte.
laden Die Anfrage läuft, noch keine Antwort. Ein Skeleton in Kartenform.
fertig Antwort angekommen und gültig. Die echte Katalogkarte.
fehler Anfrage abgelehnt oder ohne Ersatz gescheitert. Eine kurze Fehlermeldung.
useX.js, der Zustand, bevor irgendetwas anderes dazukommt 
// zustand kennt nur diese vier Werte, nie einen fünften
const zustand = ref('idle')
const karte = ref(null)
const meta = ref(null)

async function laden() {
  zustand.value = 'laden'
  try {
    const antwort = await fetchKarte()
    karte.value = antwort.data
    meta.value = { source: antwort.source, ms: antwort.ms }
    zustand.value = 'fertig'
  } catch {
    zustand.value = 'fehler'
  }
}
Kernsatz

Ein einzelnes isLoading kennt nur zwei Zustände und kann darum immer nur die Hälfte der Wahrheit sagen. Ein benannter zustand mit vier Werten sagt in jedem Moment genau eine Sache: was gerade gilt, nicht bloß, was gerade nicht gilt.

§2Der Skeleton während der Wartezeit

Der erste Zugriff auf einen Tresor-Eintrag läuft über den Netz-Pfad aus Konzept 03: echte Zeit, ungefähr ~600 ms, bevor die Antwort da ist. Genau in diesem Fenster steht zustand auf laden, und genau hier entscheidet sich, ob die Seite ruhig bleibt oder ob sie zuckt.

Ein leerer Bildschirm ist keine Antwort. Ein Layout, das bei Ankunft der Karte plötzlich seine Höhe ändert, ist auch keine. Die Komponente zeigt während zustand === 'laden' einen Platzhalter in der Form der späteren Karte: gleiche Höhe, gleiche Ränder, nur ohne Inhalt. Kommt die echte Karte an, ersetzt sie den Platzhalter, ohne dass darunter etwas springt.

Aus Konzept 03

Der erste Zugriff auf einen Eintrag ist immer der langsame Netz-Pfad, etwa ~600 ms. Jeder weitere Zugriff auf denselben Eintrag kommt aus dem Cache des Lesetischs, in etwa ~1 ms. Der Skeleton überbrückt genau die erste, langsame Wartezeit, beim zweiten Mal ist er meist zu kurz zu sehen, um überhaupt aufzufallen.

§3Der erste Wächter: AbortController

Ein Klick startet fetchKarte(). Bevor die Antwort da ist, klickt der Nutzer einen anderen Eintrag an. Die erste Anfrage läuft trotzdem weiter, im Hintergrund, obwohl niemand mehr auf sie wartet. Kommt sie doch noch zurück, schreibt sie in zustand und karte, auf Werte, die längst zu einem anderen Eintrag gehören. Verlässt der Nutzer die Ansicht ganz, schreibt am Ende sogar eine Komponente in einen ref, die es gar nicht mehr gibt.

Zwei Auslöser für abort()

Der erste Wächter ist ein AbortController. Jede Anfrage bekommt ihren eigenen: ein signal, das an fetch weitergereicht wird, und eine abort()-Methode, die genau diesen einen Lauf beendet. Startet eine neue Anfrage, bricht sie zuerst die alte ab. Verschwindet die Komponente (onUnmounted), bricht sie die noch laufende Anfrage ebenfalls ab. Der dabei entstehende AbortError ist kein echter Fehler, sondern die Bestätigung, dass der Abbruch funktioniert hat, und wird stillschweigend ignoriert.

useX.js, ein Wächter pro Anfrage 
let controller = null

async function laden() {
  controller?.abort()               // alten Lauf sofort stoppen
  controller = new AbortController()
  zustand.value = 'laden'

  try {
    const antwort = await fetchKarte({ signal: controller.signal })
    karte.value = antwort.data
    zustand.value = 'fertig'
  } catch (fehler) {
    if (fehler.name === 'AbortError') return  // gewollt, keine Meldung
    zustand.value = 'fehler'
  }
}

onUnmounted(() => controller?.abort())

Der Abbruch ist proaktiv: er spart Netzwerk und verhindert die meisten verirrten Antworten. Eine Garantie ist er trotzdem nicht. Netzwerke sind unberechenbar, und in seltenen Fällen kommt eine eigentlich abgebrochene Antwort dennoch an, kurz bevor der Abbruch greift. Ein einzelner Wächter reicht darum nicht.

§4Der zweite Wächter: das Wettrennen der Antworten

Stell dir zwei Klicks kurz hintereinander vor. Klick A trifft einen Tresor-Eintrag: langsam, ungefähr ~600 ms. Noch bevor A zurück ist, klickt der Nutzer auf B, ein Cache-Treffer, ungefähr ~1 ms. B ist so schnell, dass es garantiert zuerst antwortet.

Der Ablauf

  1. Klick auf A. laden() startet den ersten Lauf: lauf wird 1, meine merkt sich 1.
  2. A ist ein Tresor-Treffer. Die Anfrage steht, der Netz-Pfad braucht seine ~600 ms.
  3. Noch bevor A zurück ist, klickt der Nutzer auf B. laden() startet erneut: lauf wird 2, meine merkt sich 2.
  4. B ist ein Cache-Treffer und antwortet nach ~1 ms. meine (2) ist gleich lauf (2), die Antwort zählt, die Karte zeigt B.
  5. Kurz danach kommt endlich auch A an. meine (1) ist ungleich lauf (2), A gehört nicht mehr zum aktuellen Lauf.
  6. Die Antwort von A wird verworfen. zustand und karte bleiben, was B gesetzt hat.
Die Falle

Zwei Anfragen laufen gleichzeitig. Die langsame wurde zuerst gestartet, die schnelle zuletzt. Ohne Wächter gewinnt nicht die Anfrage, die zuerst gestartet wurde, und auch nicht die, die zur aktuellen Ansicht passt, sondern schlicht die, die als Letztes antwortet. Kommt die alte, langsame Antwort nach der neuen, schnellen an, überschreibt sie eine korrekte Karte mit veralteten Daten. Der Nutzer sieht dann etwas, das er gar nicht mehr angefragt hat.

Die Lösung: ein Lauf-Zähler

Der zweite Wächter braucht kein Netzwerk und keinen zusätzlichen Request. Er ist nur eine Zahl: ein Zähler, der bei jedem neuen Lauf um eins steigt, und eine lokale Kopie, die sich jeder Lauf beim Start merkt.

useX.js, der Wettlauf-Wächter 
let lauf = 0                     // zählt jede neue Anfrage hoch

async function laden() {
  const meine = ++lauf              // meine eigene Nummer, gerade gezogen
  zustand.value = 'laden'

  const antwort = await fetchKarte()

  if (meine !== lauf) return          // ein neuerer Lauf hat längst übernommen

  karte.value = antwort.data
  zustand.value = 'fertig'
}
Nach dem Fix

Mit dem Lauf-Zähler ist die Reihenfolge der Antworten egal. Es zählt nicht, wer zuerst zurückkommt, sondern wer noch zum aktuellen Lauf gehört. Die Karte zeigt immer das Ergebnis des letzten Klicks, nie ein Ergebnis, das schon überholt ist.

Der Lauf-Zähler und der AbortController lösen zusammen ein und dasselbe Problem auf zwei Ebenen: veraltete Arbeit wird gestoppt, wo es geht, und veraltete Antworten werden verworfen, wo Stoppen nicht gereicht hat.

§5Der Vertrag nach außen

Von außen sieht die composable trotzdem einfach aus. Sie gibt nur einen kleinen, immer gleichen Vertrag zurück: { zustand, karte, meta, laden }. Vier Werte und eine Funktion, mehr braucht die Komponente nicht zu kennen.

Der Vertrag der Composable

Die Komponente kennt weder fetch noch AbortController noch lauf. Sie kennt nur zustand, karte, meta und die Funktion laden, und rendert einfach, was der zustand gerade sagt. Genau wie der Server aus Konzept 01 die fremde Welt hinter einer Grenze versteckt, versteckt die composable die ganze Wächter-Logik hinter sich.

Ein kleiner Stolperstein am Rande: laden taucht hier zweimal auf, einmal als einer der vier Werte von zustand, einmal als Name der Funktion, die einen neuen Lauf startet. Zwei verschiedene Rollen, zufällig dasselbe Wort, kein Widerspruch im Code.

Diese drei Bausteine, der Zustand, die beiden Wächter, der Vertrag nach außen, sind hier nur an einem generischen useX skizziert. „Deine useArchiv baust du in Aufgabe 03."

Die Frage des Tages

„Wenn zwei Antworten um die Wette laufen, welche darf am Ende wirklich gewinnen?"

Trage diese Frage durch Aufgabe 03. Klick in deinem Test schnell zweimal hintereinander und frag nicht, welche Antwort zuerst zurückkommt, sondern welche Anfrage gerade noch gültig ist. zustand zeigt vier Gesichter, die Wächter sorgen dafür, dass zu jedem Zeitpunkt nur eines davon stimmt.

Mitnehmen

Eine composable modelliert den Zustand einer Anfrage sauber, mit vier benannten Werten statt einem einsamen Boolean. Während zustand auf laden steht, zeigt die Komponente einen Skeleton statt einer leeren Seite. Ein AbortController stoppt veraltete Anfragen, ein monotoner Lauf-Zähler verwirft veraltete Antworten, die trotzdem noch ankommen. Nach außen zeigt die composable nur einen kleinen Vertrag, und die Komponente rendert, was er gerade sagt.
